# Shopper Spectrum
### Customer Segmentation and Product Recommendation in E-Commerce

A complete data science project that cleans an online retail transaction
dataset, explores it, segments customers with RFM + KMeans clustering, builds
an item-based product recommender, and serves all of it through a styled,
multi-page Streamlit app with explicit, grouped navigation.

## 2. Setup in VS Code

1. **Open the folder** — `File > Open Folder…` and select `shopper_spectrum/`.
2. **Create a virtual environment** (recommended). Open a VS Code terminal
   (`` Ctrl+` ``) and run:

   ```bash
   python -m venv venv
   # Windows:
   venv\Scripts\activate
   # macOS / Linux:
   source venv/bin/activate
   ```

3. **Install dependencies:**

   ```bash
   pip install -r requirements.txt
   ```

4. If VS Code asks to select a Python interpreter, pick the one inside
   `venv` (bottom-right corner of the window, or `Ctrl+Shift+P` →
   "Python: Select Interpreter").

## 3. Run the Pipeline (Step by Step)

Run each script from the project's **root folder** (not inside `src/`), in
order. Each step reads the output of the previous one.

```bash
# Step 1 — clean the raw data
python src/data_cleaning.py

# Step 2 — exploratory data analysis (saves charts to outputs/eda/)
python src/eda.py

# Step 3 — RFM features + customer segmentation (KMeans)
python src/rfm_segmentation.py

# Step 4 — build the product recommendation model
python src/recommendation.py
```

Or run all four in one go:

```bash
python src/train_models.py
```

Each script prints progress and summary statistics to the terminal so you
can follow along and verify things look right before moving to the next
step.

## 4. Launch the App Locally

Once the `models/` folder has been populated (Step 3 and 4 above), start
the Streamlit app from the project root:

```bash
streamlit run app.py
```

VS Code will show a `Local URL` (usually `http://localhost:8501`) — open it
in your browser, or `Ctrl+Click` the link in the integrated terminal.

### Navigation

`app.py` builds the sidebar explicitly with `st.navigation()`, grouped into
three sections:

- **Overview**
  - 🏠 **Home** — landing page with quick links to each tool
  - ℹ️ **How It Works** — plain-English walkthrough of the whole pipeline
- **Tools**
  - 🔎 **Product Recommendation** — type a product name (e.g. `WHITE HANGING
    HEART T-LIGHT HOLDER`) and get a ranked list of 5 similar products, each
    with its similarity score
  - 👥 **Customer Segmentation** — enter Recency (days since last purchase),
    Frequency (number of orders), and Monetary (total spend) to see the
    predicted segment — **High-Value**, **Regular**, **Occasional**, or
    **At-Risk** — with a color-coded badge and a plain-English explanation
- **Analytics**
  - 📊 **EDA Dashboard** — headline metrics plus tabs for sales insights,
    customer insights, and segmentation insights

Because navigation is built explicitly with `st.Page` / `st.navigation`
(rather than relying on folder auto-discovery), the same grouping, icons,
and default page show up identically locally and once deployed.

## 5. Deploying the App

This structure works as-is on **Streamlit Community Cloud** or any other
Streamlit host:

1. Push the project to a GitHub repository (the `models/` and `data/`
   folders can be included, or regenerated by a build step that runs
   `python src/train_models.py` before the app starts).
2. On [share.streamlit.io](https://share.streamlit.io), create a new app
   and set:
   - **Repository / branch**: your repo
   - **Main file path**: `app.py`  ← important, this is what makes the
     grouped sidebar navigation and all pages show up after deployment.
3. Add `requirements.txt` (already included) so the host installs the same
   dependencies.

Because `app.py` defines every page explicitly via `st.navigation()`, the
sidebar sections, icons, and default page behave identically whether you
run it locally or after deploying — no broken links, no missing pages, and
no dependency on Streamlit's older folder-naming conventions.

## 6. How It Works

**Data cleaning** — drops rows with missing `CustomerID`/`Description`,
removes duplicates, drops cancelled orders (`InvoiceNo` starting with `C`),
and removes non-positive quantities/prices.

**RFM segmentation** — for every customer we compute:
- **Recency**: days since their last invoice
- **Frequency**: number of distinct invoices
- **Monetary**: total amount spent

These are log-transformed, standardized, and clustered with **KMeans**
(k=4, chosen to match the four business segments in the brief). Each
cluster is automatically ranked by its average RFM profile and mapped to
a segment name — the cluster with the best combination of low recency,
high frequency, and high monetary value becomes "High-Value Customers,"
down to "At-Risk Customers" for the least engaged group.

**Product recommendation** — builds a customer x product purchase matrix
(quantities), then computes **cosine similarity** between products
(item-based collaborative filtering): products bought by similar sets of
customers, in similar relative amounts, score as more similar. The app
displays the top 5 as a ranked list with similarity scores.

**Styling** — `assets/style.css` defines the color palette, card layout,
segment badges, and recommendation-list styling, loaded once in `app.py`
via `utils.load_css()` so every page looks consistent.

## 7. Re-running After Changing the Data

If you replace `data/online_retail.csv` with new data, just re-run
`python src/train_models.py` — it regenerates the cleaned CSV, charts, and
all model artifacts the app depends on.
